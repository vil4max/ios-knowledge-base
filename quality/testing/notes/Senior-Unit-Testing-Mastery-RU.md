# Unit-тесты на уровне Senior / Lead (iOS, Swift)

**Назначение:** закрыть пробел после собеса — **понимать**, **проектировать** и **писать** unit-тесты как сеньор: границы, детерминизм, моки, асинхронность, изоляция, отсутствие флейков.

**Связь с репозиторием:** общий файл темы — [Testing-Unit-UI-Snapshot.md](../Testing-Unit-UI-Snapshot.md); сеть в unit — [V/20 Q46](../../../V.%20Данные%20и%20сеть/20%20Networking%20—%20URLSession%2C%20REST%2C%20WebSocket%2C%20GraphQL%2C%20gRPC/Networking-URLSession-REST-WebSocket.md#q46).

---

## 1. Что значит «писать тесты как сеньор»

| Junior / Middle | Senior / Lead |
|-----------------|---------------|
| Пишет тест «чтобы было» | Явно решает **что не тестировать** в unit и почему |
| Дублирует прод-код в тесте | Тестирует **контракт** и **политику** (ошибки, границы, порядок побочных эффектов) |
| `sleep`, случайные задержки | **Детерминизм**: контролируемое время, фейки, `await` |
| Один огромный тест | Много **коротких** сценариев, **одна причина падения** на тест |
| Мок «на всё» | **Test double** по роли: Spy / Stub / Fake, узкие протоколы |

**Итог одной фразой:** unit-тест — это **спецификация поведения** модуля при **контролируемых** зависимостях, а не копия приложения.

---

## 2. Пирамида и границы unit

- **Unit:** чистая логика, **use case**, **mapper**, **policy** (retry, валидация), **view model** без UIKit-дерева.
- **Интеграция:** реальный `URLSession` к локальному серверу, Core Data stack, несколько модулей вместе.
- **UI:** XCUITest, снапшоты layout (если принято в команде).

**Не unit (обычно):** рендер SwiftUI/Auto Layout пиксель-в-пиксель, «нажали кнопку в VC» без выделенной логики, прохождение всего роутера если он тонкий.

---

## 3. Словарь test doubles (как на собесе)

| Термин | Роль | Проверка |
|--------|------|----------|
| **Stub** | Возвращает заранее заданные данные | Результат вызова SUT |
| **Spy** | Запоминает **вызовы** (имя метода, аргументы, счётчики) | Взаимодействие с зависимостью |
| **Fake** | Упрощённая **рабочая** реализация (in-memory store) | Поведение + иногда вызовы |
| **Mock** | В книгах — общее слово; в iOS-командах часто = **Spy** или **Stub** | Договориться в команде |

**Именование:** `UserServiceSpy`, `ClockStub`, не «`MockEverything`».

---

## 4. XCTest и Swift Testing (кратко)

- **XCTest:** `XCTestCase`, `XCTAssert*`, `XCTestExpectation` + `wait(for:timeout:)` для legacy-колбэков.
- **Swift Testing:** `@Test`, `#expect`, `#require`, параметризация; **предпочтительно** для новых таргетов на поддерживаемых версиях.

**Правило сеньора:** новый асинхронный код тестировать через **`async throws`** тесты, а не через цепочки expectations, если нет веской причины.

---

## 5. Асинхронность без флейков

**Плохо:** `sleep(1)`, `RunLoop.current.run`, «подождём пока» без явного условия.

**Хорошо:**

- `await` на **контролируемом** async API.
- Для колбэков — **обёртка** в `withCheckedContinuation` в проде или тестовый адаптер.
- **Контролируемые часы** (`Clock`, протокол `TimeProvider`, `Task.sleep` с тестовым `Clock` где доступно).

**Отмена:** после действия проверить `Task.isCancelled` / что spy зафиксировал **отмену** подписки, а не «ничего не упало».

---

## 6. MainActor и UI-логика

- Если SUT помечен **`@MainActor`**, тесты должны **`await MainActor.run`** / тест на **`MainActor`** (в зависимости от стиля), иначе **data race** в Swift 6.
- **Не** смешивать произвольный `DispatchQueue` и `@MainActor` без явной модели.

## 7. Вопросы–ответы (собес / самопроверка)

**Q1. Чем unit отличается от integration?**  
**A:** Unit — **быстро**, **без сети/диска**, **один модуль**; integration — **реальные** границы и окружение.

**Q2. Зачем протокол над сервисом?**  
**A:** Подмена **Spy/Stub** в тесте без подмены прод-типа.

**Q3. Сколько assert на тест?**  
**A:** Один **сценарий**; несколько assert допустимо, если падение однозначно читается (или разбить тест).

**Q4. Как тестировать сеть?**  
**A:** Фейк **`APIClient`** или **`URLProtocol`**; не `URLSession.shared` в глобале. См. **V/20 Q46**.

**Q5. Как тестировать время и таймеры?**  
**A:** Абстракция **`Clock` / `DateProvider`**, в тесте фиксированные моменты; не реальные `Date()` для дедлайнов.

**Q6. Что проверять в ViewModel?**  
**A:** Реакцию на **события** (входы) и **состояние** (выходы), вызовы use case **в правильном порядке**; не «UILabel.text» если это не контракт VM.

**Q7. Как избежать хрупких тестов?**  
**A:** Не завязываться на **локализованные строки** и **порядок логов**; проверять **типы ошибок** и **структуру** данных.

**Q8. Нужен ли coverage 100%?**  
**A:** Нет; важны **критические пути** и **регрессии**; coverage — **индикатор**, не цель.

**Q9. Как тестировать `Equatable` моделей?**  
**A:** `XCTAssertEqual` / `#expect` на целиком модели или на **явных** полях; для больших моделей — **снапшот** DTO или подмножество полей.

**Q10. Thread Sanitizer в тестах?**  
**A:** Включать на CI для таргетов с конкуренцией; ловит гонки, которые unit «молча» пропускает.

## 8. Задачки (сделай сам, без готового кода)

Для каждой: **Arrange–Act–Assert**, **имя теста = сценарий**, **без sleep**.

1. **Калькулятор скидки:** цена + процент; границы 0%, 100%, ошибка при >100%.
2. **Валидация email:** минимум три кейса (валид, без `@`, пустая строка).
3. **State machine заказа:** `placed → paid → shipped`; запрет недопустимых переходов; проверка `Spy` на вызовы «отправить уведомление» только из `paid`.
4. **Retry policy:** максимум 3 попытки только при `URLError.networkConnectionLost`; на 401 — без retry; Spy на количество вызовов «execute».
5. **Debouncer:** абстракция `Clock`; за фиксированным временем один вызов, при двух событиях быстрее дедлайна — один вызов (Fake clock продвигаешь вручную).
6. **Кэш с TTL:** `DateProvider`; после «истечения» — промах и повторный fetch (Spy).
7. **Mapper DTO → Domain:** nil в optional поле → дефолт в домене; отсутствие ключа → явная ошибка декодинга.
8. **Конкурентный счётчик:** два `Task` инкрементят счётчик под `actor` — итог детерминирован; тот же сценарий без actor — зафиксировать TSan (учебный эксперимент).

**Критерий «сеньор»:** для каждой задачи ты можешь объяснить, **что не тестируешь** и какой **double** выбрал.

## 9. Мини-шаблоны (идеи кода, без комментариев внутри Swift)

**Протокол + Spy (суть):**

```swift
protocol AuthAPI {
    func refreshToken() async throws -> String
}

final class AuthAPISpy: AuthAPI {
    private(set) var refreshTokenCallCount = 0
    var refreshTokenResult: Result<String, Error> = .success("token")

    func refreshToken() async throws -> String {
        refreshTokenCallCount += 1
        return try refreshTokenResult.get()
    }
}
```

**Async-тест (XCTest):**

```swift
func test_refreshToken_success_incrementsCallCount() async throws {
    let spy = AuthAPISpy()
    let sut = TokenRefresher(api: spy)
    _ = try await sut.ensureValidToken()
    XCTAssertEqual(spy.refreshTokenCallCount, 1)
}
```

**Правило:** SUT не знает, что это Spy — только протокол.

## 10. Антипаттерны (частые причины провала на собесе)

- Тест **зависит от порядка** запуска других тестов.
- **Глобальный синглтон** не сбрасывается между тестами.
- **`Date()` / `.now`** в логике без подмены → флейки на границе секунды / таймзоны CI.
- Тестируется **UIKit**, а не извлечённая логика.
- **Слишком широкий** catch — тест «зелёный», баг не пойман.
- **Нет негативных** сценариев (ошибка сети, пустой массив, `cancel`).

## 11. Чеклист ревью тестов (как лид)

- [ ] Имя теста читаетcя как **спецификация**.
- [ ] Нет **sleep** и «магических» таймаутов без константы и причины.
- [ ] Зависимости **внедрены**; нет скрытого **singleton**.
- [ ] Асинхронность через **`async`** / корректный expectation.
- [ ] Есть **как минимум один** негативный сценарий на публичный метод с ошибкой.
- [ ] Spy проверяет **и результат, и взаимодействие**, где это важно.
- [ ] Тест **быстрый** (<100 ms типично для unit без сети).
